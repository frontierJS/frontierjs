fli documentation
fli
A modular CLI automation platform where every command is a plain .md file. Drop a file, it runs. No compilation, no rebuilding. The same command files power three interfaces: CLI, this Web GUI, and a planned TUI.
how it works
When you run fli git:commit, the runtime:
  1. Scans both command roots and builds a registry
  2. Finds the .md file for git:commit
  3. Compiles it into an ESM module in memory — no temp files
  4. Validates args and flags against the frontmatter
  5. Calls run(context) with everything the command needs
two command roots
FLI scans two locations at startup and merges them into a single registry. Project commands take precedence over core commands with the same title.
rootpathlabel
fliRoot/commands/ships with fli — always availablecore
projectRoot/cli/src/routes/your project — scoped to this repoproject
key concepts

contextThe object passed to every command — paths, env, exec, git, log…

Live reload. Commands are compiled fresh on every run. Edit a .md file and your next fli <command> sees the change immediately — no restart needed.
command anatomy
Every command is a .md file with three sections: YAML frontmatter, an optional <script> block for helpers, and a fenced js block that runs on execute.
full example
--- title: git:commit description: Conventional commit prompt alias: gc examples: - fli gc - fli gc --dry args: - name: message description: Commit message required: false flags: push: type: boolean char: p description: Push after committing defaultValue: false --- <script> // helpers here — available to the js block below import { execSync } from 'child_process' const getStagedFiles = () => execSync('git diff --cached --name-only', { encoding: 'utf8' }).trim().split('\n') </script> ```js // this block runs when the command executes const staged = getStagedFiles() if (!staged.length) { log.error('Nothing staged'); return } const msg = arg.message || await question('Commit message: ') context.exec({ command: `git commit -m "${msg}"`, dry: flag.dry }) if (flag.push) context.exec({ command: 'git push', dry: flag.dry }) ```

Built-in flags. Every command automatically gets --dry (-d), --test (-t), and --step — you never need to declare them.
context api
Every command's js block runs inside a function that receives context. The most-used properties are available as top-level locals.
top-level locals
// Positional args — by name from frontmatter arg.name // e.g. arg.path, arg.query // Named flags flag.dry // always present flag.push // your custom flags // Styled logging log.info('Starting...') // blue log.success('Done') // green log.warn('Watch out') // yellow log.error('Failed') // red log.dry('Would run: ...') // cyan — use for --dry output // ZX globals echo('hello') // write to stdout await question('Name: ') // interactive prompt await `git push` // ZX shell execution
context object
// Shell execution — respects --dry automatically context.exec({ command: `git push`, dry: flag.dry }) // Resolved project paths context.paths.root // project root context.paths.api // api/ directory context.paths.web // web/ directory context.paths.db // db/ directory context.paths.schema // db/prisma/schema.prisma context.paths.webPages // web/src/routes/ context.paths.webComponents // web/src/components/ // Environment variables (from .env + process.env) context.env.PROD_SERVER context.env.ANTHROPIC_API_KEY // Shared state for _steps/ pipelines context.config.myKey = value // write in orchestrator context.config.myKey // read in steps
context.git
Git helpers are built in — no imports needed:
context.git.branch() // current branch name context.git.status() // array of changed file lines context.git.isDirty() // true if uncommitted changes context.git.lastTag() // most recent tag context.git.isAffected() // changes since last tag context.git.ahead() // commits ahead of upstream context.git.behind() // commits behind upstream context.git.log(tag) // [{hash, subject, author}] since tag context.git.remote() // origin remote URL // All methods accept an optional dir argument context.git.branch(dir) // branch in a different directory
_steps/ pipelines
Large commands break into numbered step files that share state via context.config. Steps run in sequence. The orchestrator's own body runs first, then each step in order.
directory structure
commands/npm/release/ index.md ← orchestrator: sets context.config from flags _steps/ 01-test.md ← skip: "context.config.noTest" 02-build.md ← optional: true 03-version.md 04-publish.md 05-push.md ← skip: "flag.dry"
step frontmatter
fielddescription
optional: trueFailure warns and continues to the next step
skip: "expr"JS expression — if truthy, the step is skipped. Has access to flag and context
running steps
fli release patch # all steps fli release patch --dry # step 5 skipped (skip: "flag.dry") fli release patch --step 3 # re-run only step 3
sharing state between steps
// In the orchestrator (index.md): context.config.server = env.PROD_SERVER context.config.startTime = Date.now() // In any step: const { server, startTime } = context.config context.exec({ command: `ssh ${server} "npm run deploy"`, dry: flag.dry })
Step files are never registered. The registry skips any .md file inside a _steps/ directory — they can't be run directly, only as part of their orchestrator.
add a command
There are two ways: use the scaffold command, or write the file directly. Either way, it's live as soon as it's saved.
option 1 — scaffold it
# Interactive — prompts for name, args, flags fli make:command # With a name pre-filled fli make:command myapp:sync
This creates the file in cli/src/routes/ and opens it in your editor.
option 2 — write the file
  1. Create cli/src/routes/myapp/sync.md (or any path — the title in frontmatter is the command name)
  2. Add the YAML frontmatter with title, description, and any args/flags
  3. Write a fenced js block with your logic
  4. Run fli myapp:sync — it works immediately
minimal example
--- title: myapp:sync description: Sync data from the remote API alias: sync flags: production: type: boolean description: Sync from production defaultValue: false --- ```js const url = flag.production ? context.env.PROD_API_URL : context.env.DEV_API_URL log.info(`Syncing from ${url}...`) const data = await fetch(url).then(r => r.json()) log.success(`Got ${data.length} records`) ```
project vs core
Commands in cli/src/routes/ are project-scoped — they only appear when fli is run from that project directory. To make a command available everywhere, add it to fliRoot/commands/ instead.
where the file lives vs what it's called
The file path is irrelevant. You can put myapp:sync in any file — routes/sync.md, routes/myapp/sync.md, routes/myapp/sync/index.md. The title in frontmatter is the only thing that matters for the command name.
extend fli
FLI is designed to grow with your workflow. There are three levels of extension: project commands, core namespace additions, and deeper runtime hooks.
level 1 — project commands
The simplest extension: add .md files to your project's cli/src/routes/. These commands only appear in that project. Use them for project-specific automation that doesn't belong in the shared core.
# Initialise the project routes directory fli init # Then add commands for your project cli/src/routes/ deploy/ index.md ← deploy:all (your real deploy pipeline) _steps/ 01-api.md 02-web.md seed/ index.md ← seed:all (custom data seeder)
level 2 — new core namespaces
Add a new namespace to fliRoot/commands/ to make it available across all projects. A namespace is just a folder — there's no registration step.
commands/ ai/ ← ai:ask git/ ← git:commit, git:push… myteam/ ← your new namespace standup.md ← myteam:standup deploy.md ← myteam:deploy
level 3 — extending the context
The context object is assembled in core/runtime.js. To add a helper that's available in every command, add it to the Command() function:
// In core/runtime.js, after config.git is set: config.slack = { post: async (msg) => { await fetch(process.env.SLACK_WEBHOOK, { method: 'POST', body: JSON.stringify({ text: msg }) }) } } // Now every command can do: await context.slack.post(`Deployed ${flag.target} ✓`)
the .fli.json config
Drop a .fli.json in any project root to override defaults:
{ "routesDir": "cli/src/routes", // where fli scans for project commands "defaultNamespace": "myapp", // default ns for make:command "editor": "code" // used by fli edit <command> }
